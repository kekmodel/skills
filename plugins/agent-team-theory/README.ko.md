# Agent Team Theory

*계산적 조직 이론을 에이전트 팀 설계에 적용하면 어떨까?*

조직 이론은 수십 년간 인간 팀이 왜 특정 구조를 필요로 하는지 설명해왔습니다 — 처리 용량 한계, 위임 리스크, 조정 비용, 정보 비대칭. 이것들은 인간만의 문제가 아닙니다. 구조적 문제입니다. 에이전트에게도 똑같이 발생합니다.

그런데 멀티 에이전트 시스템은 여전히 직감으로 설계됩니다: "플래너 하나, 코더 하나, 리뷰어 하나 넣자." 그 구조가 충분한지, 최소한인지, 맞기는 한 건지 아무도 묻지 않습니다.

이 플러그인은 묻습니다. Ashby, Simon, Williamson, Galbraith의 조직 이론을 생성 문법으로 형식화하고, 태스크가 실제로 요구하는 팀 구조를 **도출**합니다.

```
/team
```

태스크를 설명하세요. 플러그인이 분석하고, 구조를 도출해 보여주고, 확인을 받은 후 팀을 구성해 실행합니다.

## 조직 이론에서 에이전트 팀으로

파이프라인은 네 가지 기초 결과를 차용합니다:

- **Ashby의 필요다양성 법칙** (1956) — 팀의 대응 레퍼토리는 직면하는 교란과 맞아야 함
- **Simon의 준분해가능성** (1962; 노벨 경제학상 1978) — 태스크는 모듈로 분해되지만 잔여 상호의존성이 남음
- **Williamson의 거래비용 경제학** (1975; 노벨 경제학상 2009) — 모든 경계와 핸드오프가 구조가 흡수해야 할 마찰을 생성
- **Galbraith의 정보처리 관점** (1973) — 구조는 정보 부하와 불확실성에 대한 대응

이를 닫힌 형식 체계로 합성합니다: 7개 제1원리 → 9개 조직적 압력 → 16개 의무 → 13개 설계 패턴 → 4개 구조 원자. 파이프라인은 어떤 태스크든 이 체계를 통과시켜 최소 팀을 출력합니다.

## 동작 원리

1. **태스크 타이핑.** 12개 구조적 속성을 점검: 병렬 가능? 핸드오프에 계약 필요? 독립 검증 가능?
2. **압력 식별.** 어떤 조직적 문제가 활성화되는가? 용량 한계, 위임 리스크, 반복 예외?
3. **의무 생성.** 각 압력이 팀이 충족해야 할 구체적 요건을 생성.
4. **최소 커버 계산.** 모든 의무를 커버하는 최소 패턴 집합을 찾음. 팀이 불필요하면 그렇다고 알려줌.
5. **구성, 실행, 평가.** 자신에게 역할을 배정하고, 나머지를 스폰하고, 실행 후 구조가 효과적이었는지 평가.

## 네 가지 기본 원자

모든 팀은 같은 네 가지 원자로 구성됩니다:

| 에이전트 | 역할 |
|---|---|
| **Operator** | 실제 작업 수행. 여러 개가 병렬 또는 순차로 동작 가능. |
| **Coordinator** | 태스크 라우팅, 경계 관리, 예외 처리. 직접 실행하지 않음. |
| **Evaluator** | 검증, 선별, 비평. Operator로부터 구조적으로 독립. |
| **Rewriter** | 반복 실패 진단, 규칙 변경 제안. 필요할 때만 등장. |

그리고 **State** — 계약과 공유 컨텍스트를 담는 공유 기판. 에이전트가 아님.

메뉴에서 고른 역할 이름이 아닙니다. 이론이 제1원리에서 도출한 환원 불가능한 원자입니다.

## 예시

태스크: 레거시 결제 시스템을 새 API로 마이그레이션하면서, 라이브 트래픽은 중단 없이 유지하고, 규제 감사 로그를 남기고, 롤백 계획도 준비해라.

코딩 에이전트라면 워커 3개를 던질 겁니다. 파이프라인은 더 많은 것을 봅니다:

- 작업이 병렬 도메인에 걸침 (마이그레이션, 트래픽, 롤백) → Operator 다수
- 모듈이 서로 영향 (마이그레이션 상태가 라우팅을, 라우팅이 롤백을 결정) → 경계 관리
- 레거시 엣지 케이스 반복 → 예외 처리 + Rewriter
- 규제 컴플라이언스에 독립 검증 필요 → Evaluator
- 양방향 의존성이 있는 교차 도메인 조정 → 거버넌스 스택

도출된 구조 — 14개 의무, 9개 패턴, 8개 원자:

```
           [Coordinator]
          /       |       \
  [Operator]  [Operator]  [Operator]
  migration    traffic     rollback
          \       |       /
       [State(contract+shared)]
              |          |
  [Evaluator(verify)]  [Evaluator(monitor)]
              |
  [Coordinator(governance)]
              |
         [Rewriter]
```

Coordinator 2개 (디스패치용, 거버넌스용), 도메인별 Operator 3개, Evaluator 2개 (검증 + 컴플라이언스 모니터링), 반복 레거시 이슈를 위한 Rewriter 1개. 직감으로 설계해 보세요.

## 오케스트레이터 없음

구조를 도출한 후 플러그인은 팀 *안에서* 자신에게 역할을 배정합니다 — 보통 Coordinator. 위에서 감독하는 게 아님. Evaluator는 항상 독립적으로 스폰됩니다 (설계 선택이 아닌 구조적 요구사항).

활성화된 압력이 없으면 솔로로 실행합니다. 1인 팀은 대체가 아닌 정당한 도출 결과입니다.

## 이론이 제공하는 것

| 구성 요소 | 어떤 질문에 답하는가 |
|---|---|
| 제1원리 (P1-P7) | 구조는 왜 존재하는가? |
| 파생 압력 (D1-D9) | 구조 없이 무엇이 잘못될 수 있는가? |
| 설계 패턴 (DP1-DP13) | 어떤 구조적 대응이 가능한가? |
| 합성 계산법 | 최소 충분 팀을 어떻게 계산하는가? |
| 구조 문법 | 보편적 기본 원자로 어떻게 표현하는가? |
| 태스크 패밀리 (TF1-TF8) | 정준 구조는 어떻게 생겼는가? |
| 실패 의미론 (F1-F5) | 무엇이 왜 깨졌는가? |
| 안티패턴 (AP1-AP14) | 어떤 증상이 어떤 원인에 대응하는가? |
| 평가 (G0-S4) | 구조가 실제로 작동했는가? |

전체 이론 (~900줄): [`references/AGENT_TEAM_THEORY.md`](references/AGENT_TEAM_THEORY.md)
